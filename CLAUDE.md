# CLAUDE.md — liferay-release-analytics

This file provides guidance to Claude Code when working in this repository.

## What this is

Internal platform for Liferay release quality intelligence. Aggregates
signals from Jira, Testray, lizard, git, and feature flags into PostgreSQL
(GCP-hosted), with R-based statistical models and Looker Studio / Shiny
dashboards surfacing insights to engineering teams and leadership.

**Built and maintained by:** Release Quality Engineering
**Primary owner:** Release Team

## Four use modes

1. **Pipeline** — scheduled R/SQL jobs that extract, transform, and load
   data into fact tables (`extract/`, `transform/`, `utils/`)
2. **Dashboards** — Looker Studio (CSV exports) and Shiny Flexdashboards
   reading from pipeline outputs (`reports/`, `apps/dashboard/`)
3. **Claude Code workflows** — interactive loops that reason over platform
   data for classification and recommendation tasks (`apps/triage/`)
4. **Release operations** — human-run scripts for the Release Team's
   day-to-day: bundle validation, Jira task automation, DB upgrade
   harness, ticket-range extraction (`ops/`)

When in doubt about where new work belongs:
- Scheduled, repeatable data work → pipeline
- Visualization or reporting → dashboards
- Interactive, judgment-requiring tasks → Claude Code workflow
- Human-run release ops (bundle QA, Jira mutations, DB upgrades) → `ops/`
- Exploratory one-offs → `ad-hoc/` (safe to experiment, promote later)

## Schema ground truth

Do not guess at these — they are load-bearing and frequently wrong in
unfamiliar code:

- **Join key:** `dim_module_component_map.module_path` → `dim_module.module_path_full`
- **Defect columns:** `lpd_count` and `lpp_count` — not `lpd`, `lpp`, or `defect_count`
- **DB connection:** `get_db_connection()` with no arguments, defined in
  `config/release_analytics_db.R`. Credentials in `config/config.yml` —
  never hardcode, never commit.
- **240 components, 15 teams** — `dim_component` is the authority

## R conventions — always follow these

```r
# DB connection
source("config/release_analytics_db.R")
con <- get_db_connection()  # no arguments

# Explicit package prefixes always
dplyr::select(), dplyr::filter(), dplyr::mutate()
readr::read_csv(), readr::write_csv()

# Absolute paths from project root — never here::here()
readr::read_csv("data/churn_by_module_Q.csv")

# Upsert pattern
DBI::dbExecute(con, "
  INSERT INTO table (col1, col2) VALUES ($1, $2)
  ON CONFLICT (col1, col2) DO UPDATE SET col1 = EXCLUDED.col1
", params = list(...))

# Logging
logger::log_info("Step N: upserting {nrow(df)} rows")
```

## Complexity source

**lizard** is the complexity source (CCN + NLOC). SonarQube is retired —
do not reference it, do not restore `extract/extract_sonarqube.R`.

- Exclusions: `modules/third-party/`, `/antlr/`, `modules/dxp/apps/osb/`
- CCN cap: 100, applied in `stg_lizard_raw → fact_file_complexity`

## PR risk scoring (apps/scoring/)

Standalone tool — not part of the scheduled pipeline. Run against a
liferay-portal branch to get a composite risk score before submitting a PR.

Weights: complexity 28%, churn 25%, defects 20%, test coverage 15%,
dependencies 12%.

**`max_ccn` p95 denominator is fixed at 16.** Do not recalibrate silently —
rationale is in `apps/scoring/README.md`. Change only with explicit intent
and document the reason.

## Two-file dashboard pattern

R scripts that produce `.RData` are intentionally separate from `.Rmd`
files that load it. This avoids Jira/DB re-fetches on every dashboard
render. Do not collapse them into one file.

Example: `FixManagementAnalysis.R` → `output/fixmgmt_dashboard.RData`
→ `FixManagementDashboard.Rmd`

## Architectural decisions — why things are the way they are

**Why R for the pipeline:** LDA topic modeling, count regression, text
mining — maps naturally to R's statistical ecosystem. Outputs are CSVs;
the language choice is invisible to end users.

**Why feature-flagged churn gets a dampening weight, not zero:** Flagged
churn represents real code change activity even if the feature isn't
customer-facing yet. Zeroing it would undercount genuine risk. The
dampening factor is stored in `fact_forecast_input.churn_dampening_factor`
but not yet consumed by scoring — this is a known backlog item.

**Why `max_ccn` denominator is fixed:** Prevents silent score recalibration
if outlier components are added or removed from the dataset. Floating
denominators make trend comparisons meaningless across quarters.

**Why triage has two modes (batch + Claude Code):** The batch pipeline
(`run_triage.sh`) is fast and reproducible for large build pairs. The
Claude Code workflow is better for ambiguous cases, new routines, and
building labeled training data with richer rationale. Both write to
`fact_triage_results` with different `classifier` values so results are
comparable.

**Why `fact_triage_results` matters long-term:** It is the labeled training
data for a future NN-based PR risk prediction layer. Every BUG
classification with a `culprit_file` is a labeled outcome. Quality of
labels now determines quality of the model later.

## When to ask before acting

Stop and ask — do not proceed silently — when:
- Changing any fact table schema
- Modifying scoring denominators or weights
- Adding new credential requirements
- Creating files outside the directories listed above
- Touching `run_pipeline.sh`
- Editing `ops/upgrades/` scripts (destructive Docker ops, customer dumps)
  or `ops/release_jira/` scripts (Jira mutations, sub-task creation)

These decisions affect multiple consumers downstream.

## ops/ conventions

`ops/` contains vendored Release Team scripts — authored externally
and run by humans, not the scheduled pipeline. When editing:

- Preserve original `# Author:` and `# VERSION:` headers in the shell
  scripts under `ops/upgrades/` — bump the version when behavior
  changes.
- Preserve the MIT `LICENSE` file in `ops/utils/`.
- The Python scripts in `ops/release_jira/` use relative
  `sys.path.append('../utils')` to import `liferay_utils` — run from
  their own directory.
- Jira credentials for `ops/release_jira/` scripts live in
  `~/.jira_user/` (encrypted via `pycryptodome`), not `config/config.yml`.
- The release lead is hardcoded as `bahar.turk` in
  `ops/release_jira/release_constants.py`. Surface this if it needs
  to change; don't silently edit.

## Setup (new teammates — read this first)

When someone runs `claude` in this repo for the first time, check
whether their environment is ready by verifying these four things.
If any check fails, offer to walk through the fix interactively
before doing anything else.

### Check 1 — Database running
```bash
docker compose ps
```
Expect `release_analytics_db` with status `healthy`.
If not running: `docker compose up -d` and wait ~10 seconds.

### Check 2 — config/config.yml exists
```bash
ls config/config.yml
```
If missing: `cp config/config.yml.example config/config.yml`
Then prompt the user to fill in DB, Jira, and Testray credentials.
See `docs/SETUP.md` for the credential guide.

### Check 3 — R packages installed
```r
renv::status()
```
If packages are missing: `renv::restore()`

### Check 4 — DB connection works
```r
source("config/release_analytics_db.R")
con <- get_db_connection()
DBI::dbGetQuery(con, "SELECT 1")
```
If this fails, the most common causes are:
- Docker not running (Check 1)
- Wrong credentials in config.yml (Check 2)
- R packages not installed (Check 3)

### After checks pass
Tell the user their environment is ready and ask what they want to
work on. Suggest starting points:
- "Run me through the pipeline" → explain extract/ → transform/ → utils/
- "I want to work on triage" → open apps/triage/, load triage.skill
- "Show me the dashboards" → explain reports/ and export_looker.R flow

Full setup guide: `docs/SETUP.md`
